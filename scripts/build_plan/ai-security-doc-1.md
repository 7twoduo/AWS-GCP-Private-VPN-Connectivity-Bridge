## 6. High-Risk Issues

### 🟠 High: Hardcoded Secrets (Gitleaks)
**Severity:** 🟠 High
**Source:** Gitleaks
**Category:** Secrets
**Affected Resource:** `terraform.tfstate`, `terraform.tfstate.backup`

**Why It Matters:**
Hardcoded secrets in state files or configuration scripts can be leaked to version control systems or other unauthorized users, leading to immediate compromise of VPN credentials and shared keys.

**Recommended Fix:**
Remove all plaintext secrets from the code and state files. Use a secret management service (e.g., AWS Secrets Manager, HashiCorp Vault) and reference them via data sources.

### 🟠 High: Unrestricted Security Group Ingress
**Severity:** 🟠 High
**Source:** Checkov / Public Exposure Scan
**Category:** Network Exposure
**Affected Resource:** `Modules\AWS-VM\main.tf`

**Why It Matters:**
Allowing `0.0.0.0/0` on all ports or specific management ports (like ICMP) exposes the infrastructure to brute force attacks and scanning from any location on the internet.

**Recommended Fix:**
Restrict `cidr_blocks` in security group rules to specifically known internal IP ranges or VPN gateway CIDRs.

### 🟠 High: Sensitive Port Exposed to Network
**Severity:** 🟠 High
**Source:** Checkov / Public Exposure Scan
**Category:** Network Exposure
**Affected Resource:** `Modules\AWS-VM\main.tf`

**Why It Matters:**
Exposing port 22 (SSH) or other sensitive management ports to the entire network/internet increases the risk of unauthorized access and lateral movement.

**Recommended Fix:**
Restrict SSH access to a specific bastion host or an internal VPN range.

## 7. Detailed Remediation Guide

### SEC-001 — Hardcoded Secrets
**Source:** Gitleaks
**What to Change:** Replace `shared_secret` and `preshared_key` values with references to secret management lookups.
**Why It Matters:** Prevents credentials from being stored in plain text within the repository or state files.

### SEC-002 — Unrestricted Security Group Ingress
**Source:** Checkov / Public Exposure Scan
**What to Change:** Update `cidr_blocks` from `["0.0.0.0/0"]` to specific internal CIDRs.
**Why It Matters:** Reduces the attack surface by limiting who can attempt to connect to your instances.

### SEC-005 — EC2 Instance Has Public IP
**Source:** Checkov / KICS
**What to Change:** Set `associate_public_ip_address = false` for internal instances.
**Why It Matters:** Prevents non-public-facing resources from being reachable directly from the internet.

### SEC-007 — Using Default Service Account
**Source:** KICS / Checkov
**What to Change:** Define a specific `service_account` with an explicit email for GCP instances.
**Why It Matters:** Limits the blast radius of a compromised instance by restricting its permissions to only what is necessary.

## 8. Sample Code

### Problem Example (SEC-002)
```hcl
# Vulnerable: SSH and ICMP open to everyone
resource "aws_security_group" "vm_sg" {
  ingress {
    description = "SSH"
    from_port   = 22
    to_port     = 22
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
}
```

### Corrected Example (SEC-002)
```hcl
# Secure: SSH restricted to internal VPN CIDR
resource "aws_security_group" "vm_sg" {
  ingress {
    description = "SSH"
    from_port   = 22
    to_port     = 22
    protocol    = "tcp"
    cidr_blocks = ["10.0.0.0/8"] # Example internal range
  }
}
```
